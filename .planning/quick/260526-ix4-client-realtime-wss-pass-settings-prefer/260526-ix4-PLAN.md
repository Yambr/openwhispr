<objective>
Forward the user's `settings.preferredLanguage` (normalized to ISO-639-1 base code via `getBaseLanguageCode`) end-to-end from renderer through the main-process realtime IPC to the WSS URL as `?language=<code>`. The server (xgohagty's v1.0.9, shipping in parallel) reads this query param, validates against its whitelist `['en','ru']`, and injects `transcription.language` into the OpenAI session.update frame. Closes language drift in realtime dictation (and meeting mic/system streams).

Purpose: Without this passthrough, realtime transcription falls back to OpenAI auto-detect — short Russian utterances frequently get mis-detected as English. The renderer already plumbs `language` through 3 of 4 hops; only the final WSS URL construction is missing. This task closes the gap and fixes a renderer-side normalization bug (lines 2065/2290 in audioManager.js leak `en-US` instead of `en`, which the server whitelist rejects).

Output:
- `src/helpers/openaiRealtimeStreaming.js` — destructures `language` from options; URL appends `&language=<encoded>` when truthy
- `src/helpers/ipcHandlers.js` — `connectInner` (line ~5133) forwards `language: options.language` into `streaming.connect`
- `src/helpers/audioManager.js` — lines 2065 + 2290 use `getBaseLanguageCode(preferredLang)` instead of the inline `&& !== "auto"` guard
- `src/stores/meetingRecordingStore.ts` — line ~139 OpenAI-fallback return path includes `language`
- `test/helpers/openaiRealtimeStreaming.test.js` — existing Vitest suite extended with 7 new language-suffix assertions on URL build (no new test file, no `package.json` change — see revision note below)
</objective>
