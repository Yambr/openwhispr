# WR-01 — HOST-02 test hooks shipped in production renderer (bypass SSRF gate)

**Severity:** WARNING (MEDIUM in practice — mitigated by contextIsolation)
**Status:** ✅ RESOLVED 2026-06-03 — quick task 260603-r0p, commit be1cb424.
Gated the three hooks behind `window.electronAPI?.isE2E === true` (preload
bridges runtime `NODE_ENV==="test"`). LIVE-PROVEN via CDP: same prod dist →
hooks `undefined` when NODE_ENV unset, present under NODE_ENV=test. See
`.planning/quick/260603-r0p-.../LIVE-VERIFICATION.md`. Original plan/analysis
below.
**Surfaced by:** code review of commit eb9716d4 (#8 fix)
**Origin:** fork-only, commit ec92c536 (HOST-02), NOT in upstream.

## Finding

`src/lib/auth.ts:152-166` registers three test hooks on `window`
UNCONDITIONALLY in every renderer load:

- `window.authClientBaseUrlForTest()` — reads resolved baseURL
- `window.__authClientForTest` — the live authClient
- `window.__zustand_setServerUrl(url)` — **writes `serverUrl` directly to the
  Zustand store via `setServerUrl`, bypassing `ServerUrlField.validate()`**

The #8 fix (eb9716d4) makes `serverUrl` determine the OIDC sign-in HOST. The
fix's SSRF-safety argument rests on `serverUrl` only ever holding a value that
passed `ServerUrlField`'s HTTPS-only + RFC1918/loopback/link-local screen
(M2/WARN-02). `__zustand_setServerUrl` is an in-renderer path that sets
`serverUrl` to ANY `string | null` with no validation — directly undermining
that guarantee for the auth host.

## Risk

Capability amplification in production. An attacker who achieves renderer
code-execution (malicious renderer bundle / compromised dependency) can call
`window.__zustand_setServerUrl("https://attacker.example")`, then a subsequent
OIDC sign-in would deep-link the OAuth flow + callback to the attacker host —
an auth/credential redirect.

**Practical precondition:** `contextIsolation: true` on all BrowserWindow
configs means web content cannot reach the renderer `window` global, so this
requires renderer-code-execution (not a remote web attacker). That bounds it to
WARNING/MEDIUM rather than HIGH. But the hook is pure unnecessary attack
surface in a shipped build, and it weakens an auth-host guarantee.

## Suggested fix (bundler-aware — do NOT rush)

Gate the entire `if (typeof window !== "undefined") { ...hooks... }` block so
the hooks exist ONLY in e2e/test builds, never in production.

**Correct gate signal:** the e2e harness launches Electron with
`NODE_ENV: "test"` (`tests/e2e/fixtures/electron-launch.ts:51`). Production
builds are not `test`. So the hooks' consumers (`tests/e2e/steps/
host-runtime-override.steps.ts`, `onboarding-serverurl-email.steps.ts`) keep
working iff the gate is true under `NODE_ENV=test`.

**CAUTION (the bundler trap, cf. memory rolldown_tree_shake):** `auth.ts` is
renderer code bundled by Vite (`src/vite.config.mjs`). A bundled renderer does
NOT automatically see `process.env.NODE_ENV` — Vite must `define` it, or you
use `import.meta.env.MODE`/`import.meta.env.DEV`. Before landing the gate:
  1. Confirm HOW the e2e renderer is built (dev server vs production bundle) and
     what `import.meta.env.MODE` / `process.env.NODE_ENV` actually evaluate to
     INSIDE the e2e renderer at runtime — add a temporary `console.log` and run
     one `@host` e2e scenario.
  2. Pick the gate that is `true` in e2e and `false` in `npm run build` prod.
     Likely candidates: a Vite `define` of `process.env.NODE_ENV` keyed off the
     launch env, or a dedicated `VITE_OPENWHISPR_E2E` flag set in the e2e build
     step. Mirror whatever mechanism `main.js`'s NODE_ENV==='test' auth-token
     gate already relies on.
  3. Re-run the full `@host` e2e feature (host-runtime-override) AFTER gating —
     the hooks MUST still be reachable there, and MUST be absent in a prod
     build (verify by grepping the built `dist/` bundle for
     `__zustand_setServerUrl`).

This is deferred from quick task 260603-ogm specifically because a naive
`import.meta.env.DEV` guard would silently break the host-override e2e suite,
and getting it right needs the bundler verification above — out of scope for a
2-line auth-host fix.

## Acceptance

- Production renderer bundle contains NO `__zustand_setServerUrl` /
  `authClientBaseUrlForTest` / `__authClientForTest` (grep built `dist/`).
- `@host` e2e suite (host-runtime-override.feature) still green.
- `serverUrl` is unsettable from any in-renderer path that skips
  `ServerUrlField` validation, in production.
