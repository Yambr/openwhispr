  - Scenario 0: default — no env vars.
  - Scenario 1: OPENWHISPR_OAUTH_GOOGLE=false.
  - Scenario 2: OPENWHISPR_OAUTH_APPLE=false.
  - Scenario 3: OPENWHISPR_OAUTH_MICROSOFT=false.

After each build, grep `dist/` recursively for each target literal. Note: `signInWithSocial("google")` after rolldown minification may become `signInWithSocial("google")` still (rolldown preserves literal string args), but the variable `signInWithSocial` itself may be renamed. Use `grep -F` (fixed string, no regex) and accept that the grep is on the literal. If `signInWithSocial` is mangled in production builds, fall back to greppingfor the *call-site literal* (e.g., `"google"` appearing in a context like `(provider="google"`). Empirically: the i18n keys (`auth.social.continueWithGoogle` etc.) are unlikely to be minified and serve as a robust target.

Realistic interpretation: bundle-grep targets that survive minification reliably are:
  - Domain literals (URLs in fetch / new URL): always preserved.
  - i18n key strings (passed to t()): always preserved.
  - Component class names like "GoogleIcon": only if exposed via `displayName` or kept by sourcemaps; otherwise unreliable.

Pragmatic mitigation in the script: for each target, run TWO greps:
  (a) raw target string against `dist/**/*.{js,html,css}`.
  (b) fallback against the corresponding i18n key (e.g., for `GoogleIcon`, also grep `continueWithGoogle`).
Pass = ALL grep results return 0 (for disabled provider) or ≥ 1 in any of the (a)/(b) variants (for enabled provider).

Build duration: vite build for this app runs ~30-60s. 4 sequential builds = ~2-4 min for `npm run verify:oauth-gating`. Document this in the script header so the runner knows to expect it.
