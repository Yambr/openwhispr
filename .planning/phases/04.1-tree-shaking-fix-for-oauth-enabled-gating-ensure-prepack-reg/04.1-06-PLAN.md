       **"Feature Gating Mechanism"** (~30-50 lines):
       - Explain why tree-shake is required (binary size + audit surface).
       - Document the three-layer pattern:
         a. **Build-config generation** — `scripts/generate-build-config.js` BOOL_DEFAULTS +
            `resolveBool` parses env, emits `OAUTH_*_ENABLED` and feature flags as literal
            booleans in `src/config/build-config.generated.{ts,cjs}`.
         b. **Renderer DCE** — `src/config/defaults.ts` re-exports `Generated.X_ENABLED`
            directly (no indirection). Consumers wrap mounting points in
            `{X_ENABLED && <SubComponent />}`. Sub-component split is required when the
            gated subtree has hooks (rules-of-hooks) or contains imports of large modules.
         c. **Main + preload** — `src/helpers/ipcHandlers.js` wraps registration blocks in
            `if (BuildConfig.X_ENABLED) { ... }`. `preload.js` uses
            `...(BuildConfig.X_ENABLED ? { /* methods */ } : {})` conditional spread.
       - Reference verify scripts: `npm run verify:oauth-gating` (4 scenarios for OAuth
         providers) and `npm run verify:feature-gating` (4 scenarios for feature flags).
       - Note: the previous `pickBool` indirection in `defaults.ts` was inert — Vite `define`
         only substitutes static `import.meta.env.X` references. Phase 04.1 removed it.
       - Cross-link: `scripts/verify-pack-regenerates-build-config.js` (CFG-08 regression test).
