## Fixed Issues

### CR-01: process.exit(1) inside runBuild skips cleanup

**Files modified:** `scripts/verify-oauth-gating.js`
**Commit:** 7d3ce89
**Applied fix:** Replaced both `process.exit(1)` calls inside `runBuild()`
(after `genResult.status !== 0` and `buildResult.status !== 0`) with
`throw new Error(...)`. Wrapped the scenario loop in `try/catch/finally`
so a thrown error is captured into `runError`, the `finally` block still
runs `runBuild("restore-default", {})` to leave the dev tree at its
default config, and only after cleanup does the script exit non-zero
via `process.exit(1)`. Verified end-to-end via `npm run verify:oauth-gating`
(passed: 4 scenarios, 63 greps, 0 violations) — confirms the cleanup
mechanism still gates correctly under the new throw-based control flow.

### CR-02: startOAuth has no catch — failures silently swallowed

**Files modified:** `src/components/GoogleCalendarSection.tsx`
**Commit:** 19082b3
**Applied fix:** Added a `catch (err)` block to the `startOAuth` callback
that logs the failure via `console.error("[GoogleCalendarSection] OAuth
failed:", err)`. Spinner state still resets via the existing `finally`.
Toast surfacing was considered but not added — keeping the fix scoped
to the CodeRabbit-flagged issue (silent swallow). The toast UX is a
separate enhancement.

### CR-03: handleDisconnect ignores gcalDisconnect return value

**Files modified:** `src/components/GoogleCalendarSection.tsx`
**Commit:** 6a19d1f
**Applied fix:** Capture the IPC result and gate UI removal on
`result?.success !== false` (mirroring the `startOAuth` pattern).
On a soft-failure response (`{ success: false }`), the account stays
visible in the UI; the existing `onGcalConnectionChanged` listener
will resync if the server later confirms removal. Also added a
`catch` block paralleling CR-02.

### CR-04: BACKEND_SPEC.md base URL points at pre-refactor env path

**Files modified:** `docs/BACKEND_SPEC.md`
**Commit:** 183182e
**Applied fix:** Rewrote the Base URL row in the Conventions table.
Verified actual env-var reality before writing (per workflow
instructions, did not blindly trust CodeRabbit's claim):

- `OPENWHISPR_BACKEND_URL` is the new build-time canonical export from
  `src/config/defaults.ts` (re-exported from `build-config.generated.ts`,

- `VITE_OPENWHISPR_API_URL` (legacy) STILL exists and feeds
