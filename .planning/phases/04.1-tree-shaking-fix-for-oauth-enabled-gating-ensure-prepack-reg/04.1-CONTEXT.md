# Phase 04.1 Context

## Goal

Fix the OAuth gating tree-shake bug discovered during Phase 04 smoke testing AND extend the same gating pattern to three new corporate-minimal feature flags (CFG-09): `OPENWHISPR_BILLING_ENABLED`, `OPENWHISPR_REFERRALS_ENABLED`, `OPENWHISPR_STREAMING_ENABLED` — all defaulting to `false` per the 2026-05-08 project pivot.

## Requirements addressed

- **CFG-07** — OAuth gating physically tree-shakes disabled provider code paths from the renderer bundle
- **CFG-08** — `npm run pack` and `npm run build` both regenerate `build-config.generated.{ts,cjs}` so env-var overrides actually take effect (currently only `prebuild` runs the generator — `prepack` does not)
- **CFG-09** — Three new feature flags, all default `false`, each tree-shake-verified:
  - `OPENWHISPR_BILLING_ENABLED` (Stripe checkout/portal/switch-plan + `/api/stripe/*`)
  - `OPENWHISPR_REFERRALS_ENABLED` (referral stats/invite UI + `/api/referrals/*`)
  - `OPENWHISPR_STREAMING_ENABLED` (AssemblyAI/Deepgram WebSocket live ASR + `/api/{streaming,deepgram-streaming}-token`)

## Bug evidence (Phase 04 smoke test, 2026-05-08)

### Bug 1: prepack does not regenerate build-config

`package.json` defines:
- `prebuild` script — chains `compile:native` → all `download:*` → **`node scripts/generate-build-config.js`**
- `prepack` script — chains the same downloads but **does NOT call `generate-build-config.js`**
- `pack` script — runs `build:renderer` then `electron-builder --dir` (its lifecycle hook is `prepack`, not `prebuild`)

Consequence: `OPENWHISPR_OAUTH_GOOGLE=false npm run pack` does not propagate the env var into `src/config/build-config.generated.ts` because the generator never runs. The previously committed value persists.

### Bug 2: tree-shake doesn't actually remove gated code

After running the generator manually (`OPENWHISPR_OAUTH_GOOGLE=false node scripts/generate-build-config.js`) and rebuilding:
- `src/config/build-config.generated.ts` correctly emits `export const OAUTH_GOOGLE_ENABLED = false;`
- BUT `npm run verify:oauth-gating` still fails with `google-disabled: target "gcalStartOAuth" expected absent, found: dist/assets/IntegrationsView-*.js`
- The string `gcalStartOAuth` is the renderer-side method name on `window.electronAPI`. It still appears in the bundled `IntegrationsView-*.js` because the Phase 04 WR-01 fix gates UI display behind `if (OAUTH_GOOGLE_ENABLED)` but does not isolate the `window.electronAPI?.gcalStartOAuth?.()` call site behind a top-level conditional that the bundler can dead-code-eliminate.

Likely root cause hypothesis (to validate during execution):
- Vite/Rolldown tree-shaking only removes branches when the gate constant is a **literal** at the consuming module's top level. The current pattern reads `Generated.OAUTH_GOOGLE_ENABLED` from a module export object — minifier may not constant-fold across the import boundary, especially when the symbol is referenced inside an arrow function passed to a JSX `onClick`.
- Alternative: the `gcalStartOAuth` symbol leaks through because preload.js exposes it unconditionally on `window.electronAPI`, even when the renderer code never calls it. preload.js is bundled separately from the renderer.

## Implementation hints (for planner)

- For **CFG-08** (prepack regenerates): simplest fix is to either (a) move generator into `prepack` chain too, or (b) add explicit step in `pack` script before `electron-builder`. Prefer (b) — most explicit, least surprising.
- For **CFG-07** (tree-shake fix): investigate three approaches in this order, pick the one that works:
  1. **Vite `define`** — re-emit booleans as Vite `define` constants (literal substitution at parse time, guaranteed dead-code-eliminated). Currently they may go through `import { OAUTH_GOOGLE_ENABLED } from '@/config/build-config.generated'` which is a runtime export.
  2. **Inline gate at call site** — wrap every `window.electronAPI?.gcalStartOAuth?.()` AND every `window.electronAPI.gcalDisconnect`/`onGcalConnectionChanged` call with `if (OAUTH_GOOGLE_ENABLED) { ... }` inline so the closure itself is dead.
  3. **preload.js conditional exposure** — gate `gcalStartOAuth` registration in preload.js so it's never exposed when disabled. Note: preload runs in main-process context and reads env at app start; needs the generated `.cjs` flavour.
- For **CFG-09**, mirror the OAuth gating pattern. Each new flag:
  - Add to `BOOL_DEFAULTS` in `scripts/generate-build-config.js` (default `false`)
  - Add export in `src/config/defaults.ts`
  - Wrap all UI mounting points (Stripe checkout buttons, referral modal, streaming toggles) behind the gate using whichever approach worked for OAuth
  - Wrap IPC handlers in `src/helpers/ipcHandlers.js` behind the gate (handlers should not register when disabled — main-process side)
  - Add to `verify-*-gating` script with default-build expecting symbols ABSENT

## Risks

- **Risk: Vite `define` requires literal at config-parse time, but the generator runs before vite.config.mjs imports the .cjs.** Need to verify that `loadEnv()` + reading the generated cjs synchronously in vite config works. Phase 03 already does `define: { ... buildTimeDefaults }` — confirm the pattern works for booleans.
- **Risk: Existing renderer code may reference `window.electronAPI.gcalStartOAuth` in multiple files.** A single file fix won't cover all symbols. Plan must do a `grep -r "gcalStartOAuth\|stripeCheckout\|referralStats" src/` audit before implementing.
- **Risk: Streaming code is bigger than expected** — `useChatStreaming` is 141kB minified, suggests significant logic. Gating may require splitting into "non-streaming chat" + "streaming chat" modules to tree-shake cleanly. Plan should investigate before committing to a specific approach.
- **Risk: Phase 04 verify-oauth-gating's default-build scenario expects OAuth symbols PRESENT.** When we change defaults, we mustn't regress that scenario.
