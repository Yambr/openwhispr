### CR-01 (Warning, Major) â€” `scripts/verify-oauth-gating.js:137-156`

**`process.exit(1)` inside `runBuild` skips cleanup in caller's `finally` block**

`runBuild()` is called from a try-block where the `finally` runs `runBuild("restore-default", {})` to restore the dev tree to default config. When `runBuild` itself calls `process.exit(1)` on build failure, Node terminates **without running the finally**, leaving `src/config/build-config.generated.{ts,cjs}` and `preload-*.generated.cjs` in whatever state the failing scenario set them to.

**Impact:** After a CI failure or local dev run that fails mid-scenario, the developer's local tree has wrong flags committed-feeling. `git status` shows nothing wrong (files are gitignored) but the next dev session starts from an unexpected baseline.

**Fix:** Replace `process.exit(1)` with `throw new Error(...)` inside `runBuild`. The caller's try-catch + finally will then run cleanup, then re-throw or process.exit at the outer level.

Lines: 141 (genResult.status check) and 156 (buildResult.status check).

### CR-02 (Warning, Major) â€” `src/components/GoogleCalendarSection.tsx:49-74`

**`startOAuth` has no `catch` â€” OAuth failures silently swallowed**

`startOAuth` uses `try/finally` (no `catch`). Exceptions from `window.electronAPI?.gcalStartOAuth?.()` propagate to `handleConnect`, which also has no try/catch. An unhandled rejection from a button's `onClick` handler is silently dropped â€” spinner stops, user sees nothing.

Violates CLAUDE.md guideline: "Use try/catch blocks for error handling in async operations, IPC calls."

**Fix:** Add `catch (err) { console.error("[GoogleCalendarSection] OAuth failed:", err); }` block. Also consider surfacing user-facing toast.

### CR-03 (Warning, Major) â€” `src/components/GoogleCalendarSection.tsx:76-88`

**`handleDisconnect` ignores `gcalDisconnect` return value â€” UI removes account on soft failure**

`startOAuth` correctly gates state update behind `result?.success && result.email`. But `handleDisconnect` calls `gcalDisconnect` then *unconditionally* removes the account from UI regardless of response. If IPC handler resolves with `{success: false}` instead of throwing, the account disappears from UI while still being connected server-side. No `onGcalConnectionChanged` event fires to correct it.

**Fix:** Mirror startOAuth pattern:

```ts
const result = await window.electronAPI?.gcalDisconnect?.(email);
if (result?.success !== false) {
  const current = useSettingsStore.getState().gcalAccounts;
  setGcalAccounts(current.filter((a) => a.email !== email));
}
```

Also add try/catch like CR-02.
