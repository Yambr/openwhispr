<objective>
Fix CFG-07: OAuth gating does not currently tree-shake. Phase 4's onboarding `AuthenticationStep`
gating works (DCE removes the disabled provider button), but the Settings → Integrations panel
(`IntegrationsView.tsx`) renders the Google Calendar card unconditionally, and its three
`window.electronAPI?.gcal*?.()` call sites leak the literal strings `gcalStartOAuth`,
`gcalDisconnect`, `onGcalConnectionChanged` into the bundle even when `OAUTH_GOOGLE_ENABLED=false`.

Additionally, per Phase 4 review WR-02: the current `pickBool` indirection in `defaults.ts` is
inert. Real DCE relies entirely on the imported `Generated.OAUTH_*_ENABLED` const — which is
already a literal `export const X = true|false` in the generated module. The locked decision
for this plan (see `<interfaces>` block) is **NOT** to switch to Vite `define`; instead we
reuse the proven Phase 4 AuthenticationStep mechanism: **component-split + literal
`Generated.*` re-export from `defaults.ts`**.

Finally, even with the renderer fix, `preload.js` currently exposes `gcalStartOAuth` /
`gcalDisconnect` / `gcalGetConnectionStatus` / `onGcalConnectionChanged` unconditionally on
`window.electronAPI`. Those literal strings end up in the preload bundle and partially defeat
the goal of "disabled provider code paths physically removed" (CFG-07). Mirroring the pattern
PLAN-03/04 will use for billing/referrals, this plan also gates the OAuth methods in
`preload.js` via a conditional spread on `BuildConfig.OAUTH_*_ENABLED`.

Purpose: Establish the canonical tree-shake mechanism for the entire phase. PLAN-03/04/05
copy this exact pattern for BILLING / REFERRALS / STREAMING.

Output:
- `src/components/IntegrationsView.tsx` extracts a `GoogleCalendarSection` sub-component
  containing all gcal hooks + IPC calls; mounts it via `{OAUTH_GOOGLE_ENABLED && <... />}`.
- `src/config/defaults.ts` cleaned up per WR-02 (remove inert `pickBool` indirection; keep
  literal `Generated.OAUTH_*_ENABLED` re-exports).
- `scripts/verify-oauth-gating.js` `GOOGLE_TARGETS` includes `gcalStartOAuth`, `gcalDisconnect`,
  `onGcalConnectionChanged` in absence-checks for the google-disabled scenario, AND the script
  also greps preload bundle output (not just renderer `dist/assets/*.js`) for the same targets.
- `preload.js` gates OAuth method exposures behind `BuildConfig.OAUTH_*_ENABLED`.
- All 4 verify-oauth-gating scenarios (default, google-disabled, apple-disabled, microsoft-disabled) PASS.
</objective>

**Locked mechanism decision (do NOT switch to Vite `define`):**

The Phase 4 AuthenticationStep gating works because the gate is placed at the JSX-mounting
level (`{OAUTH_GOOGLE_ENABLED && (...)}`), and the gated subtree contains the
`signInWithSocial("google")` call. Rolldown can DCE the entire JSX subtree when the gate is
`false`. The boolean value reaches that gate via a literal `export const X = true|false` in
`build-config.generated.ts`, re-exported as a same-shape literal const from `defaults.ts`.
This is sufficient for DCE — proven by AuthenticationStep working.

**The IntegrationsView fix is therefore purely about applying the same JSX-gate pattern
there, NOT changing the underlying mechanism.**

We will:
1. Extract a `GoogleCalendarSection` sub-component in IntegrationsView containing all gcal
   hooks (`useState`, `useCallback`, `useEffect` registering `onGcalConnectionChanged`) and
   IPC call sites (`gcalStartOAuth`, `gcalDisconnect`).
2. Mount it via `{OAUTH_GOOGLE_ENABLED && <GoogleCalendarSection ... />}`.
3. Remove the inert `pickBool` code from defaults.ts (per WR-02) — it's misleading.
4. Keep the existing direct `Generated.OAUTH_*_ENABLED` re-exports (lines 75-77 already correct).

This is simpler than Vite `define` and matches the existing working pattern.

**Fallback if mechanism fails empirically.** After Task 2 GREEN, run
`grep -RF gcalStartOAuth src/dist/` with `OPENWHISPR_OAUTH_GOOGLE=false`. If the string is
still present (i.e., component-split + literal re-export did NOT eliminate `gcalStartOAuth`
from the renderer bundle), the fallback is to gate the corresponding
`window.electronAPI.gcalStartOAuth` call sites with `if (OAUTH_GOOGLE_ENABLED) {...}` inline
guards. This forces dead-branch detection at the call site itself, not just at the JSX
mounting site. Document the fallback (if used) in the SUMMARY so PLAN-03/04/05 know which
pattern to apply.
