# Phase 04.1 Plan 04: REFERRALS_ENABLED build flag (CFG-09 partial 2/3) Summary

## Tree-Shake Mechanism (refines PLAN-02 for PLAN-05)

For ANY new build-time boolean flag whose gated literals live inside a UI **component subtree** (not inside an always-imported hook):

1. **Generator** — `scripts/generate-build-config.js#BOOL_DEFAULTS` adds the flag with default value, plus `emitPreloadFoo(resolved, outPath)` mirroring `emitPreloadGcal`/`emitPreloadBilling`/`emitPreloadReferrals`.
2. **Defaults re-export** — `src/config/defaults.ts` extends the direct named re-export.
3. **Sub-component file** — extract the gated UI into a dedicated `*.tsx` file. **Use STATIC import of any modules that carry the IPC literals; do NOT use React.lazy() inside the sub-component.** React.lazy() causes Vite/Rolldown to emit an orphan chunk for the dynamic import target even when the sub-component file itself is dropped from the static import graph.
4. **Parent JSX gate** — `{FOO_ENABLED && <FooEntry ... />}`. Rolldown propagates the literal across the named-re-export boundary, drops the JSX, then the import, then the sub-component file, then everything downstream of it.

For preload-side gating use the now-mature code-generated factory pattern (third instance after gcal + billing): `emitPreloadFoo` writes `preload-foo.generated.cjs`; preload.js spreads `...buildFooApi(ipcRenderer)` inside `contextBridge.exposeInMainWorld`.

For the always-imported-hook shape, use the PLAN-03 Vite-alias mechanism instead.

## Deviations from Plan

### 1. Mechanism deviation (renderer DCE — first attempt failed)

**Plan says:** "Wrap each [JSX usage of ReferralModal/ReferralDashboard] with `{REFERRALS_ENABLED && (...)}`" — and the plan-text under `<interfaces>` proposes that "Rolldown DCE WILL drop unused imports IF the only usage is inside a `false && ...` branch ... Prefer this simpler approach first."

**Reality:** Two complications.

(a) The original `ControlPanel.tsx` JSX mounted `<ReferralModal>` via `React.lazy(() => import("./ReferralModal"))` declared at module top level. Wrapping the JSX in `{REFERRALS_ENABLED && ...}` did NOT drop the dynamic-import expression because the `React.lazy()` declaration itself is unconditional and creates the chunk-emission edge during parsing. Even after extracting the lazy declaration into a sub-component file (`ReferralEntry.tsx`), the chunk was STILL emitted as an orphan: Vite/Rolldown unconditionally emits a chunk for every `import("...")` expression it parses, regardless of whether the containing module is reachable.

(b) Switching ReferralEntry to a STATIC import of ReferralModal resolved (a) cleanly: with no dynamic-import expression anywhere in the chain, the chunk is only emitted if some reachable module statically references it. When the parent gate evaluates to literal false, ReferralEntry is unused → static import to ReferralModal is unused → ReferralModal chunk is not emitted at all.

**Adopted:** Static import inside ReferralEntry.tsx. This is a hard refinement of PLAN-02's "sub-component must live in a SEPARATE file" rule: **the sub-component file must NOT contain a dynamic `import(...)` expression to the gated module.** Logged in "decisions" so PLAN-05 follows this.

### 2. Scope (ReferralDashboard.tsx)

**Plan files_modified lists:** `src/components/ReferralDashboard.tsx`.

**Reality:** ReferralDashboard.tsx is imported only by ReferralModal.tsx. When ReferralModal is dropped from the build graph, ReferralDashboard is unreachable and dropped automatically. No explicit edit needed; the file is unchanged.

### 3. ReferralModal.tsx itself

**Plan files_modified lists:** `src/components/ReferralModal.tsx`.

**Reality:** Same as above — no edit needed. The file is unchanged on disk; it simply isn't reachable from the build graph when REFERRALS_ENABLED=false.
