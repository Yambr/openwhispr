<objective>
Add `OPENWHISPR_STREAMING` build flag (default `false`, per 2026-05-08 corporate-minimal pivot)
that physically removes AssemblyAI / Deepgram WebSocket live ASR + `/api/{streaming,deepgram-streaming}-token`
IPC and the entire `useChatStreaming` 141-kB module path from the binary when disabled.

**RISK:** This plan is the most complex of the three CFG-09 flags. `useChatStreaming.ts` is
141 kB minified and is referenced from exactly 3 renderer files (`src/components/chat/ChatView.tsx`,
`src/components/AgentOverlay.tsx`, `src/hooks/useEmbeddedChat.ts` — confirmed by grep
2026-05-08). Streaming code may not cleanly tree-shake via simple JSX gating because:
1. `useChatStreaming` is a custom hook — must be called unconditionally per rules-of-hooks.
2. The hook's exports (`useChatStreaming`, `streamingToken`, `deepgramStreamingToken`) may
   be imported at module scope in consumers even when never called → those imports survive DCE.

The investigation step (Task 1) is mandatory before implementation; it determines whether
JSX-gate + sub-component-split is sufficient or whether module-split (separate streaming-only
chunk) is required. Task 1 must produce an empirical measurement table (see Task 1 below) —
not just analytical reasoning.

Purpose: Third and final CFG-09 flag.

Output:
- `STREAMING_ENABLED` boolean added to BOOL_DEFAULTS.
- Re-export in `defaults.ts`.
- `verify-feature-gating.js` extended with streaming targets.
- All consumer files apply the chosen gating mechanism (may differ per consumer based on
  Task 1 investigation).
- IPC streaming handlers (assemblyai-* + deepgram-* + token endpoints) wrapped in
  `if (BuildConfig.STREAMING_ENABLED)`.
- Preload conditionally exposes streaming methods.
- Default build: chat UI works without streaming (existing non-streaming chat path is
  the fallback).

**Tree-shake mechanism — investigation required:**
- Option A: JSX-gate consumer subtrees + ensure all streaming imports are inside gated
  sub-components. This worked for OAuth/billing/referrals but those features each had a
  single localized UI surface. Streaming spans 3 consumers.
- Option B: Module-split. Move `useChatStreaming.ts` into a lazy-loaded module and use
  `if (STREAMING_ENABLED)` to choose between streaming and non-streaming chat at
  consumer level. Rolldown's `import()` lazy chunks ARE tree-shaken when the dynamic
  import call site is dead. Requires more refactoring.
- Option C: Re-export shim. Create `src/components/chat/chatStreamingProvider.ts` that
  exports either the real `useChatStreaming` or a no-op stub based on STREAMING_ENABLED.
  Consumers always import the shim. The shim's export is conditional on a literal
  boolean (`export const useChatStreaming = STREAMING_ENABLED ? RealHook : NoopHook`).
  Rolldown can DCE the unused branch and drop the entire `useChatStreaming.ts` import
  if the literal is `false`.

Task 1 chooses among these via empirical measurement (see Task 1 action). Default
preference: Option C (shim) — least disruptive to consumers, single point of control.

**Non-streaming fallback:**
Chat already supports non-streaming responses (the regular Anthropic/OpenAI/Gemini providers
DO NOT require WebSocket streaming — those are for ASSEMBLYAI/DEEPGRAM live transcription).
Confirm during Task 1: when `STREAMING_ENABLED=false`, the chat experience falls back to
regular non-streaming providers. **No user-visible regression in default build.**

For dictation: live transcription via AssemblyAI/Deepgram is STREAMING-only. With
`STREAMING_ENABLED=false`, dictation falls back to whisper.cpp / OpenAI Whisper file-upload
mode — which is the corporate-minimal default per the pivot. This matches CONTEXT.md
constraints.
