1. **`# Application Architecture`** — top-of-doc h1.

2. **Front matter** (3–5 sentences): purpose of this doc, audience (external implementer / OSS contributor — same as SELF_HOSTING.md per D-02), how to read it (each topic has diagram + prose + key files with `file:line` citations per D-03/D-04), and a note that `docs/BACKEND_SPEC.md` and `docs/OAUTH_SPEC.md` cover the wire-level contract.

4. **`## Process Model`** — the four processes:
   - Main process (`main.js`) — Electron lifecycle, native OS integration, IPC handlers, DB, file I/O, hotkey registration, sidecar spawning.
   - Renderer process(es) — `src/main.jsx` → `src/AppRouter.jsx`; URL-based routing (`?panel=true` → ControlPanel, `?agent=true` → AgentOverlay, default → dictation overlay). Context isolation enabled.
   - Preload (`preload.js`) — `contextBridge.exposeInMainWorld("electronAPI", {...})`; the only bridge.
   - ONNX utility worker (`src/workers/onnxWorker.js`) — spawned lazily by `src/helpers/onnxWorkerClient.js`; hosts all `onnxruntime-node` inference (text embeddings, speaker embeddings, fbank); native crashes (e.g., ORT `bad_alloc`) confine to the worker; respawn with backoff; stopped in `will-quit`.
   Include an ASCII block diagram showing the 4 processes and IPC arrows (Main ⇄ Preload ⇄ Renderer; Main ⇄ ONNX Worker via parent-child message passing). Cite `main.js:1`, `preload.js:1`, `src/main.jsx:1`, `src/workers/onnxWorker.js:1`, `src/helpers/onnxWorkerClient.js:1`.

5. **`## IPC Surface`** — categorized summary per D-06 (NOT exhaustive table). Group ~150+ channels by domain prefix: `db-*` (database), `transcribe-*` (transcription pipeline), `get-*-key` / `save-*-key` (secret storage), `window-*` (window control), `hotkey-*` (hotkey registration), `meeting-*` (meeting detection), `cloud-*` (cloud reasoning passthrough), plus a short list of others (audio, paste, OAuth, model management). For EACH category include: contract pattern (one-line description), example channel name, args/return shape (one line, types only). End the section with: **"For the authoritative full list of channels, see `preload.js`."** Note the renderer-direct vs. main-process-via-IPC duality for OpenWhispr cloud calls (Phase 1 finding). Cite `preload.js`, `src/helpers/ipcHandlers.js`, and `CLAUDE.md §IPC Patterns` for the kebab-case domain-prefixed naming convention.

6. **`## Secret Storage`** — per D-08, MUST explicitly cover:
   - The 12 `SECRET_KEYS` from `src/helpers/environment.js` — list them by name (the 7 BYOK API keys + 5 enterprise cloud creds).
   - Per-key encrypted file convention: `userData/secure-keys/{KEY_NAME}.enc`.
   - OS keychain backing via Electron `safeStorage` → Keychain (macOS) / DPAPI (Windows) / libsecret (Linux).
   - Linux-without-keyring fallback: plaintext (Electron default) — explicitly call this out as a known limitation.
   - Loading: keys decrypted into `process.env` at startup by `EnvironmentManager.init()`.
   - IPC surface for renderer access: `get-{key}` reads, debounced `save-{key}` writes.
   - **Explicit constraint:** build-time env vars (Phase 3 introduces these) are NEVER for secret material — they are for *defaults and endpoints*. User-provided secrets stay runtime-only via safeStorage. (Quote PROJECT.md constraint.)
   Cite `src/helpers/environment.js:{line of SECRET_KEYS}`, `src/helpers/ipcHandlers.js` for `get-*-key`/`save-*-key` handlers.

7. **`## Model Registry`** — `src/models/modelRegistryData.json` is single source of truth (cloud + local providers). `src/models/ModelRegistry.ts` is the TS wrapper. Derived configs: `src/config/aiProvidersConfig.ts` (AI_MODES), `src/utils/languages.ts` (REASONING_PROVIDERS). Local model features: `hfRepo`, `promptTemplate`, download URL construction. 8 inference providers (`anthropic`, `enterprise`, `gemini`, `groq`, `lan`, `local`, `openai`, `openwhispr`) implementing `InferenceProvider` interface from `src/services/ai/inferenceProviders/types.ts`. 4 inference scopes (`dictationCleanup`, `dictationAgent`, `noteFormatting`, `chatIntelligence`) defined in `src/config/inferenceScopes.ts`. Cite all paths with `:line` where applicable.

8. **`## Transcription Pipeline`** — diagram of the audio pipeline: MediaRecorder API → Blob → ArrayBuffer → IPC → temp file → engine → result → temp file deleted. Engines: local whisper.cpp (`src/helpers/whisper.js`, `src/helpers/whisperServer.js`), NVIDIA Parakeet via sherpa-onnx (`src/helpers/parakeet.js`, `src/helpers/parakeetServer.js`), cloud OpenAI Whisper API. Custom dictionary integration as Whisper `prompt` parameter (cite `src/helpers/audioManager.js`, `src/helpers/whisperServer.js`). Provider preference in `LOCAL_TRANSCRIPTION_PROVIDER` and `PARAKEET_MODEL` persisted to `.env`. Models stored in `~/.cache/openwhispr/{whisper,parakeet}-models/`.

Sidecars to enumerate: whisper-server, llama-server, sherpa-onnx (Parakeet), qdrant, ONNX worker (note: not in resources/bin — spawned from `src/workers/`), windows-key-listener.exe, windows-mic-listener.exe, macos-mic-listener (Swift), macos-globe-listener (Swift), macos-fast-paste, linux-fast-paste.
