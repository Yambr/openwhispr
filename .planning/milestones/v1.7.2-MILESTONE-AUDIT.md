## v1.8.0 Prerequisite Health Check

The new milestone introduces runtime user-editable Server URL on onboarding. Prerequisites:

### A. OPENWHISPR_BACKEND_URL as single source of truth — **WEAK** *(downgraded)*

⚠ Integration-check (INT-02) overturned my initial PASS. Two parallel env-var channels carry the backend host:
- `OPENWHISPR_BACKEND_URL` (Phase 3 declared SoT) — consumed only by generator/WSS, `main.js` webRequest, `openaiRealtimeStreaming.js`. **Renderer never imports it.**
- `OPENWHISPR_API_URL` (from `VITE_OPENWHISPR_API_URL`, `src/config/constants.ts:116`) — what the renderer actually uses: `AuthenticationStep.tsx`, `EmailVerificationStep.tsx`, `src/lib/auth.ts`, and **26 call sites in `ipcHandlers.js`** via `getApiUrl()`.

CI papers over the split by setting both vars to `vars.VITE_OPENWHISPR_API_URL` (`release.yml:114-117,128-131`). **Convention, not architecture.**

This does not block closing v1.7.2 (the build works because CI sets both), but v1.8.0 MUST collapse to one SoT before adding the runtime override. Otherwise the user types `acme.com` and 26 call sites continue hitting the build-time default.

### B. Build-time gate pattern (corporate-minimal Stripe/Referrals style) — **PASS**
- `PROVIDER_LOCKDOWN_ENABLED` is the established blueprint. Direct named re-export from `build-config.generated` (DCE-safe, per `[[rolldown_tree_shake]]` lesson).
- 6 working flags as precedent: OAUTH_GOOGLE/APPLE/MICROSOFT, BILLING, REFERRALS, STREAMING, PROVIDER_LOCKDOWN.
- `verify:provider-lockdown` gate proves tree-shake actually removes literals from bundle (2 scenarios × 20 greps = 40 assertions, 0 violations).
- v1.8.0's new `OPENWHISPR_ALLOW_CUSTOM_HOST` slots into this exact pattern.

### C. Better Auth dynamic base URL — **FAIL** *(upgraded from WEAK)*

⚠ Integration-check (INT-01) confirmed this is harder than I assumed.
- `src/lib/auth.ts:12` calls `createAuthClient({ baseURL: AUTH_URL })` at module load.
- `AUTH_URL = import.meta.env.VITE_AUTH_URL` is build-time-only via Vite `define`.
- The exported `authClient` is a **frozen module-singleton** — every consumer in the renderer captures the same instance at first import.

Runtime URL reconfiguration (the v1.8.0 core feature) **cannot work** without refactoring this. Options:
  - Convert `authClient` to a lazy factory function that re-resolves baseURL from persisted settings on each call.
  - Or maintain a mutable proxy that re-creates the inner client when settings change.

This does NOT block closing v1.7.2 — the build works fine for the build-time-baked URL. But v1.8.0 Phase 1 must address it BEFORE any onboarding UI work, otherwise the new Server URL field will silently still hit the compiled-in default.
