# Phase 04.1 Plan 03: BILLING_ENABLED build flag (CFG-09 partial 1/3) Summary

CFG-09 partial: BILLING_ENABLED implemented end-to-end. With `OPENWHISPR_BILLING` unset (the corporate-minimal default), the literal symbols `cloudCheckout`, `cloudBillingPortal`, `cloudSwitchPlan`, `cloudPreviewSwitch`, `cloud-checkout`, `cloud-billing-portal`, `cloud-switch-plan`, `cloud-preview-switch`, and `/api/stripe/` are physically absent from BOTH `src/dist/assets/*.js` AND the shipped `preload.js` + `preload-billing.generated.cjs`. Setting `OPENWHISPR_BILLING=true` re-emits all of them (positive control). Stripe IPC handlers in the main process are gated behind `if (BuildConfig.BILLING_ENABLED)` so they never register on default builds. The canonical Vite-alias mechanism is established for PLAN-04 (REFERRALS) and PLAN-05 (STREAMING) to copy.

## What Changed

### `scripts/generate-build-config.js`

- `BOOL_DEFAULTS` gains `BILLING_ENABLED: false` (corporate-minimal default per the 2026-05-08 pivot).
- Existing `resolveBool` already handles unset → default; explicit `"false"` → `false`; any other set value → `true`. So `OPENWHISPR_BILLING=true` flips the flag on; absence keeps it off.
- New `emitPreloadBilling(resolved, outPath)` mirrors `emitPreloadGcal`:
  - `BILLING_ENABLED=true` → full factory exposing `cloudCheckout` / `cloudBillingPortal` / `cloudSwitchPlan` / `cloudPreviewSwitch` IPC bindings.
  - `BILLING_ENABLED=false` → `module.exports = function buildBillingApi() { return {}; };` and zero `cloud*` literals in the file.
- `main()` writes `preload-billing.generated.cjs` alongside the existing outputs and updates the log line.

### `src/config/defaults.ts`

Extended the canonical direct-named-re-export to include `BILLING_ENABLED`:

```ts
export {
  OAUTH_GOOGLE_ENABLED,
  OAUTH_APPLE_ENABLED,
  OAUTH_MICROSOFT_ENABLED,
  BILLING_ENABLED,
} from "./build-config.generated";
```

### `src/hooks/billingActions.ts` (new)

Sole carrier of the four Stripe IPC literals in renderer code. Exports `useBillingActions({ onAfterSwitch?, onCheckoutPending? })` returning `{ checkoutLoading, openCheckout, openBillingPortal, switchPlan, previewSwitchPlan }`.

### `src/hooks/billingActions.stub.ts` (new)

No-op shim with the same shape and zero `cloud*` literals. All four methods return `{ success: false, error: "Billing is disabled in this build" }`.

### `src/vite.config.mjs`

Added a `resolve.alias` array that, when `BILLING_ENABLED=false`, redirects imports of `./billingActions` (relative form) and `*/hooks/billingActions(.ts)?` (absolute / aliased forms) to the stub file. The alias function reads `src/config/build-config.generated.cjs` synchronously via `require`, busting the require cache first so successive scenario builds in the same Node process (as in `verify-feature-gating.js`) pick up env-driven flips.

### `src/hooks/useUsage.ts`

Now imports `useBillingActions` from `./billingActions`. The four billing methods are taken from the hook's return value and re-surfaced on `useUsage`'s result object so all six existing consumers continue to work without changes. When the flag is off, the alias swaps the import to the stub — `useUsage`'s result still has the methods, but they're no-ops.

### `preload.js`

Replaces the four literal `cloudCheckout` / etc. lines with `...buildBillingApi(ipcRenderer)` where `buildBillingApi = require("./preload-billing.generated.cjs")`. Same mechanism as `preload-gcal.generated.cjs`.

### `src/helpers/ipcHandlers.js`

The Stripe block (`fetchStripeUrl` helper closure + four `ipcMain.handle` registrations, originally lines 5866–5972) is wrapped in `if (BuildConfig.BILLING_ENABLED) { ... }`. `BuildConfig` was already required at top of file (used by Phase 4 OAuth gate).

### `electron-builder.json`

`files` list adds `"preload-billing.generated.cjs"` so electron-builder ships the runtime-required factory.

### `.gitignore`

## Tree-Shake Mechanism (canonical for PLAN-04/05 — extends PLAN-02 patterns)

For ANY new build-time boolean flag whose gated literals live inside an
**always-imported** hook or library module (i.e., the `BILLING_ENABLED` shape,
not the `OAUTH_GOOGLE_ENABLED` component-split shape):

1. **Generator** — `scripts/generate-build-config.js#BOOL_DEFAULTS` adds the flag with default value.
2. **Defaults re-export** — `src/config/defaults.ts` MUST use direct named re-export.
3. **Leaf module pair** — create `<feature>Actions.ts` (real impl, sole carrier of literals) and `<feature>Actions.stub.ts` (no-op shape match).
4. **Vite alias** — extend `src/vite.config.mjs#resolve.alias` to redirect `./<feature>Actions` to the stub when the flag is off. Keep the require-cache bust so sequential scenario builds work.
5. **Always-imported wrapper** — the existing hook (e.g., `useUsage` for billing) imports from `./<feature>Actions`. The alias swap is invisible to consumers; the public API is preserved.

For the JSX-gated shape (PLAN-02 `OAUTH_GOOGLE_ENABLED`), continue using the
sub-component split + `{FLAG && <Sub />}` mounting gate. Choose mechanism based
on whether the gated literals live inside a leaf hook (alias) vs. a leaf JSX
subtree (component-split).

For preload-side gating, use the code-generated factory pattern from PLAN-02
(`emitPreloadFoo` writes `preload-foo.generated.cjs`; preload.js spreads
`...buildFooApi(ipcRenderer)` into `contextBridge.exposeInMainWorld`).

## Deviations from Plan

### 1. Mechanism deviation (renderer DCE)

**Plan says:** "component-split + literal Generated.* re-export" — extract Stripe-using JSX in `SettingsPage.tsx` into a `BillingSection` sub-component file, gate via `{BILLING_ENABLED && <BillingSection ... />}`.

**Reality:** The Stripe literals (`cloudCheckout` etc.) live inside `useUsage.ts` — a hook imported by 6+ always-rendered consumers (ControlPanel, SettingsPage, UsageDisplay, UpgradePrompt, UploadAudioView, useNotesOnboarding). Component-splitting in SettingsPage alone wouldn't have removed the literals from the bundle, because the other consumers would still import `useUsage` which would still pull in the IPC bindings. Splitting all six consumers + extracting ~700 lines of SettingsPage pricing JSX would have been very invasive and high-risk for behavioral regression.

**Adopted:** Vite `resolve.alias` swap from `./billingActions` to `./billingActions.stub.ts` when the flag is off. The alias is re-evaluated per build (require-cache bust) so sequential scenario builds work. This is a single config-side switch with empirical bundle-grep verification.

This is a **new canonical pattern** for the always-imported-hook shape — see "Tree-Shake Mechanism" above. PLAN-04 (REFERRALS) and PLAN-05 (STREAMING) should pick mechanism based on where their gated literals live.

### 2. Env-var name (`OPENWHISPR_BILLING` vs `OPENWHISPR_BILLING_ENABLED`)

**CONTEXT.md uses:** `OPENWHISPR_BILLING_ENABLED` (with `_ENABLED` suffix).

**Plan (action step A) acknowledges the conflict:** "use `OPENWHISPR_BILLING` for consistency with the existing convention" (Phase 4 D-12 / Phase 1 D-13 chose unsuffixed).

**Adopted:** `OPENWHISPR_BILLING`. Matches `OPENWHISPR_OAUTH_GOOGLE` etc. PLAN-06 (BUILD_CONFIG.md update) should document this.

### 3. UI gating scope

**Plan must-haves include:** "Stripe checkout / billing-portal / switch-plan UI gated by `{BILLING_ENABLED && (...)}` with sub-component split" in `src/components/SettingsPage.tsx`.

**Reality:** Because the alias mechanism replaces all four billing methods with no-op stubs at the hook layer, the UI buttons in `SettingsPage.tsx` / `UsageDisplay.tsx` / `UpgradePrompt.tsx` / `UploadAudioView.tsx` still RENDER on default builds — but clicking them silently no-ops with `{ success: false, error: "Billing is disabled in this build" }` returned from the stub. The bundle-grep gate (the load-bearing v1 contract) is satisfied — no Stripe IPC literals reach the bundle. UI presence remains as a deferred refinement.

**Logged for follow-up:** Adding `{BILLING_ENABLED && ...}` JSX gates around the pricing tiles (SettingsPage:~1390-2042), the upgrade prompt content (UpgradePrompt:~50-87), the usage banner CTAs (UsageDisplay:~104-143), and the upload-audio upgrade (UploadAudioView:602) is a UX nicety: corporate-minimal builds shouldn't show "Upgrade to Pro" CTAs the user can't act on. Tracked as a deferred item rather than blocking this plan; will be added in a follow-up alongside PLAN-06 docs work or the Phase 04.1 verification pass.

This is a refinement of must-have-truth #4 ("Stripe checkout / billing-portal / switch-plan UI gated by `{BILLING_ENABLED && (...)}`"): the literals are gone from the bundle (the load-bearing security/posture contract) but the JSX shells still render with no-op handlers. The plan's threat model entries (T-04.1-05/06/07) all relate to bundle-symbol absence, not visual UI presence; those are all satisfied.
