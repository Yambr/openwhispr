# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-08
**Depth:** standard
**Files Reviewed:** 19 (across 6 plans + 1 phase-level config)

## Summary

Phase 04.1 closes CFG-07/08/09 with an impressive, well-documented tree-shaking architecture: named re-exports through `defaults.ts` (literal propagation), JSX sub-component splits (PLAN-02/04), Vite alias swap-to-stub for always-imported leaves (PLAN-03/05), and code-generated preload factories (all four flags). The verify scripts are the load-bearing security contract and they are mechanically correct — the empirical bundle-grep evidence in each plan summary confirms the gates work end-to-end.

Findings below are all defense-in-depth refinements. None bypass an existing tree-shake gate or introduce a security regression. The 3 warnings flag (a) a runtime-error UX gap when `STREAMING_ENABLED=false` is combined with a user-configured streaming model in the renderer, (b) a missed env-var sanitisation in the pack-regen test which could cause flaky behavior on a developer machine with persistent `OPENWHISPR_*` exports, and (c) a misleading source comment that documents the OPPOSITE of what the code now does (PLAN-04 explicitly removed `React.lazy()` from the gated path; the comment still describes the lazy form).

The Phase 04 WR-01 fix (the original OAuth tree-shake bug) was genuinely addressed — not just suppressed. The named re-export change in `src/config/defaults.ts` (lines 82-97) is the actual root-cause fix, replacing the `import * as Generated; export const X = Generated.X` form which silently degraded to runtime checks. The empirical verification (`gcalStartOAuth: 0` in dist + preload across 4 OAuth scenarios) confirms the fix is structural rather than cosmetic.

## Warnings

### WR-01: `shouldUseStreaming()` does not consult `STREAMING_ENABLED` — confusing hard error in default build if user has realtime model configured

**File:** `src/helpers/audioManager.js:1976-2000` (`shouldUseStreaming`) + `src/helpers/audioManager.js:2242-2265` (`startStreamingSession` error path)

**Issue:** When `STREAMING_ENABLED=false` (corporate-minimal default), `STREAMING_PROVIDERS` is aliased to the Proxy stub whose `start()` returns `{ success: false, error: "Streaming disabled in this build" }` (no `code` field). However, `shouldUseStreaming()` itself does NOT short-circuit on the build flag — it returns `true` whenever a user has e.g. `cloudTranscriptionModel: "gpt-4o-transcribe"` and is signed in. The flow then enters `startStreamingSession`, the stub's `start()` returns `{success:false}` with no `code`, and the code at line 2263 throws `new Error("Streaming disabled in this build")` which surfaces to the user as a recording failure (NOT a graceful fallback to file-upload transcription).

PLAN-05 SUMMARY § "Did not gate the postServerToken..." asserts: "audioManager.shouldUseStreaming() returns false because all providers fall through the stub Proxy." This is incorrect — `shouldUseStreaming` does not look at the provider object; it only checks settings. The Proxy stub is transparent to the gate.

In the corporate-minimal pivot the path is reachable today only if (a) the user is signed in to a self-hosted OpenWhispr cloud, AND (b) sets `cloudTranscriptionModel` to a realtime model. Per PROJECT.md the default backend is configurable, so this is a real configuration users can hit.

**Fix:** Add an explicit gate in `shouldUseStreaming` so the renderer's intent matches the bundle's capability:

```js
import { STREAMING_ENABLED } from "../config/defaults";
// ...
shouldUseStreaming(isSignedInOverride) {
  if (!STREAMING_ENABLED) return false;   // <-- new
  const s = getSettings();
  if (s.useLocalWhisper) return false;
  // ...rest unchanged
}
```

This is an inline literal-boolean gate (Rolldown DCE-friendly) and turns the "streaming disabled" error into a silent fallback to file-upload transcription, matching the documented graceful-degradation intent. The stub modules can stay as defense-in-depth.

### WR-02: `verify-pack-regenerates-build-config.js#regenerateDefaults()` only sanitises `OPENWHISPR_OAUTH_*` env vars — leaves `OPENWHISPR_BILLING/REFERRALS/STREAMING` ambient if developer has them exported

**File:** `scripts/verify-pack-regenerates-build-config.js:50-66`

**Issue:** The cleanup path strips only OAUTH_-prefixed env vars before regenerating the default-state build config:

```js
for (const key of Object.keys(cleanEnv)) {
  if (key.startsWith("OPENWHISPR_OAUTH_")) {
    delete cleanEnv[key];
  }
}
```

A developer who has `export OPENWHISPR_BILLING=true` (or `_REFERRALS`, `_STREAMING`) in their shell — common during local feature work — will have those flags persist through `regenerateDefaults()`, leaving the dev tree with `BILLING_ENABLED=true` etc. after the test "cleans up." The test's pre-condition (`OAUTH_GOOGLE_ENABLED=true`) still passes, so the test silently produces a non-default state and the dev's next `npm start` runs against the wrong configuration.

Not a security issue (the test runs locally), but it violates the test's own contract ("On exit, the test ALWAYS restores `build-config.generated.{ts,cjs}` to defaults" — file header comment lines 14-15).

**Fix:** Strip ALL `OPENWHISPR_*` env vars in the cleanup pass — there is no reason to preserve any during a defaults-restore:

```js
for (const key of Object.keys(cleanEnv)) {
  if (key.startsWith("OPENWHISPR_")) {
    delete cleanEnv[key];
  }
}
```

This generalises trivially as future flags are added, eliminating drift.

### WR-03: Misleading comment in `ControlPanel.tsx` — describes a `React.lazy()` flow that PLAN-04 explicitly REMOVED

**File:** `src/components/ControlPanel.tsx:50-56`

**Issue:** The comment block above the lazy-loaded views reads:

```tsx
// Phase 04.1 PLAN-04: ReferralEntry is a thin sub-component that internally
// `React.lazy()`-imports ReferralModal. When REFERRALS_ENABLED is the literal
// false at the JSX gate `{REFERRALS_ENABLED && <ReferralEntry ... />}`,
// Rolldown propagates the literal across the named-re-export boundary and
// drops both the JSX and the (now unused) static import to ReferralEntry —
// which in turn drops the dynamic `import("./ReferralModal")` chunk and the
// referral IPC literals.
```

Two contradictions with the actual code (and with PLAN-04 SUMMARY, deviation #1):

1. "ReferralEntry…internally `React.lazy()`-imports ReferralModal" — false. `ReferralEntry.tsx` line 2 does a STATIC import (`import ReferralModal from "./ReferralModal";`). PLAN-04 SUMMARY explicitly documents the `React.lazy()` form was tried first and produced an orphan chunk; the static form was the GREEN fix.
2. "drops the dynamic `import("./ReferralModal")` chunk" — false. There is no dynamic import; the static import edge is what gets dropped.

A future contributor reading this comment will think the canonical pattern is `React.lazy()` and re-introduce the bug PLAN-04 fixed.

**Fix:** Update the comment to describe the actual mechanism (static import + parent JSX gate), and explicitly call out the "no React.lazy()" rule from the PLAN-04 decision log:

```tsx
// Phase 04.1 PLAN-04: ReferralEntry uses a STATIC import of ReferralModal
// (NOT React.lazy() — see PLAN-04 SUMMARY deviation #1). When the parent gate
// `{REFERRALS_ENABLED && <ReferralEntry ... />}` evaluates to literal `false`,
// Rolldown propagates the literal across the named-re-export boundary
// (`src/config/defaults.ts`), drops the JSX, drops the unused import to
// ReferralEntry, and the entire downstream static graph (including
// ReferralModal and the `getReferralStats` / `sendReferralInvite` /
// `getReferralInvites` IPC literals) becomes unreachable and is not emitted.
// Do NOT switch ReferralEntry to React.lazy() — Vite/Rolldown emits an orphan
// chunk for any `import(...)` expression encountered during parsing,
// regardless of whether the containing module is reachable.
```

The verify script's STREAMING_TARGETS only enumerates AssemblyAI + Deepgram literals + the two `/api/*-streaming-token` URL fragments — so a regression that re-introduces an OpenAI-realtime literal into the bundle on default builds would NOT be caught by the gate.

Empirically the alias swap removes the entire `streamingProviders.js` content (proven by the bundle-size diff in PLAN-05 SUMMARY), so this is not a current bug — it's a missing safety net.

**Fix:** Append the OpenAI-realtime symbols to `STREAMING_TARGETS`:

```js
const STREAMING_TARGETS = [
  // ...existing...
  "dictationRealtimeStart",
  "dictationRealtimeWarmup",
  "dictationRealtimeStop",
  "onDictationRealtimePartial",
];
```

### IN-02: Source-file comments in gated sub-components reference the gated literals verbatim

**File:** `src/components/GoogleCalendarSection.tsx:1-11` (refers to `gcalStartOAuth`, `gcalDisconnect`, `onGcalConnectionChanged`)

**Issue:** The header comment of `GoogleCalendarSection.tsx` lists the literal IPC method names. When OAUTH_GOOGLE_ENABLED=true the file is bundled and these literals appear in dist. When OAUTH_GOOGLE_ENABLED=false the file is tree-shaken so the literals never reach dist (which is why verify-oauth-gating passes). Today this is harmless. BUT: if a future Vite/Rolldown change preserves a leading comment of an "unreachable but kept" module (e.g., for source maps in some `legalComments` mode), the bundle-grep gate would flip from PASS to FAIL because the COMMENT contains the literal.

PLAN-02 SUMMARY locked-in decision #4 explicitly scrubbed `preload.js` comments of literal method names for this reason. The same hygiene should apply to source-side gated files.

**Fix:** Replace literal method names in source comments with descriptive prose:

```tsx
// Phase 04.1 CFG-07: Google Calendar UI + IPC call sites are isolated into a
// dedicated module so Rolldown DCE can fully eliminate this file's contents
// (including all `gcal*` IPC method literals) from the renderer bundle when
// OAUTH_GOOGLE_ENABLED is the literal `false`.
```

Apply the same to `src/hooks/billingActions.ts`, `src/components/chat/useChatStreaming.stub.ts`, `src/helpers/streamingProviders.js`, `src/helpers/streamingProviders.stub.js` — all of which currently reference gated literals in their leading comments.

### IN-04: Resolver semantics asymmetry — `OPENWHISPR_BILLING=anything` enables billing, not just `=true`

**File:** `scripts/generate-build-config.js:84-91` (`resolveBool`)

**Issue:** `resolveBool` treats any value other than the literal string `"false"` as truthy:

```js
return process.env[envKey] !== "false";
```

So `OPENWHISPR_BILLING=0`, `OPENWHISPR_BILLING=no`, `OPENWHISPR_BILLING=` (empty string with `=`), `OPENWHISPR_BILLING=disabled` all ENABLE billing. A developer reading `verify-feature-gating.js#SCENARIOS` (which uses `"true"`) might reasonably assume the inverse `"false"` path is `"true"`.

This is documented in `docs/BUILD_CONFIG.md` per PLAN-06 SUMMARY decision log ("Documented the resolver semantics asymmetry explicitly"), so it is technically intentional. But as an info-level finding: the asymmetry is a footgun for anyone scripting CI invocations and is inconsistent with the convention most env-flag tools use (`0`/`false`/empty all = false).

**Fix (optional, larger change):** Treat any truthy-string set as enabling, but accept `"0"` and `""` as also-false:

```js
const v = process.env[envKey];
if (v === undefined) return BOOL_DEFAULTS[boolKey];
return v !== "false" && v !== "0" && v !== "";
```

If kept as-is, ensure the public docs example uses `=true` consistently (it does).

### IN-05: `streamingProviders.stub.js` returns the same `STUB_PROVIDER` object reference for every key — provider identity comparisons would fail silently

**File:** `src/helpers/streamingProviders.stub.js:26-33`

**Issue:** The Proxy returns the same `STUB_PROVIDER` reference on every `get`, regardless of key. If anywhere in the code there is `provider === STREAMING_PROVIDERS.deepgram` (referential identity check) or a `Map<provider, state>` pattern, the stub would conflate distinct providers. A grep through the codebase suggests no current code path does this (`getStreamingProvider()` only invokes the methods), so this is informational.

**Fix:** None needed currently. If providers ever gain identity-based dispatch, the Proxy `get` handler should construct a fresh object per key, e.g. `get(_, key) { return { ...STUB_PROVIDER, _name: key }; }`.
