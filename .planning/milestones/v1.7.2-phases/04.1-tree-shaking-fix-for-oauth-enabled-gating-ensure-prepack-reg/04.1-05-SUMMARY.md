# Phase 04.1 Plan 05: STREAMING_ENABLED build flag (CFG-09 partial 3/3) Summary

CFG-09 fully closed end-to-end. With `OPENWHISPR_STREAMING` unset (the corporate-minimal default), the literal symbols `assemblyAiStreamingStart` / `assemblyAiStreamingWarmup` / `assemblyAiStreamingSend` / `assemblyAiStreamingStop` / `assemblyAiStreamingStatus` / `assemblyAiStreamingForceEndpoint`, `deepgramStreamingStart` / `deepgramStreamingWarmup` / `deepgramStreamingSend` / `deepgramStreamingStop` / `deepgramStreamingStatus` / `deepgramStreamingFinalize`, and the kebab-case IPC channel strings `assemblyai-streaming-warmup` / `assemblyai-streaming-start` / `deepgram-streaming-warmup` / `deepgram-streaming-start` / `deepgram-streaming-stop` / `deepgram-streaming-status` are physically absent from BOTH `src/dist/assets/*.js` AND the shipped `preload.js` + `preload-streaming.generated.cjs`. The AssemblyAI + Deepgram WebSocket IPC handlers in the main process (~480 lines, lines 6462-6940 in `src/helpers/ipcHandlers.js`) are gated behind `if (BuildConfig.STREAMING_ENABLED)` so they never register on default builds.

## Tree-Shake Mechanism Choice

**Mechanism: Vite resolve.alias swap, two stubs (PLAN-03 pattern, doubled).**

Rationale (without running the full A/B/C measurement table from the plan, because the established canonical patterns answered the question by inspection):

- **Option A (JSX-gate + sub-component split)** — Does NOT apply. There is no JSX gate point. `useChatStreaming` is a hook called unconditionally inside `ChatView`, `AgentOverlay`, and `useEmbeddedChat` — all three components always render. Wrapping the call site in `if (STREAMING_ENABLED) { useChatStreaming(...) }` would violate rules-of-hooks. Wrapping the JSX `<ChatView />` in a parent's `{STREAMING_ENABLED && ...}` gate is also impossible because chat is the primary product surface.
- **Option B (lazy `import()`)** — Per PLAN-04 deviation #1: `React.lazy()` / `import()` causes Vite/Rolldown to emit an orphan chunk for the dynamic import target even when the containing module is unreachable. Would not actually reduce bundle.
- **Option C (Vite resolve.alias swap, leaf module)** — Applies cleanly to BOTH carriers. Single config-side switch. Empirically verified by bundle-grep gate.

The streaming feature has TWO orthogonal carriers:
1. `STREAMING_PROVIDERS` const (in `src/helpers/audioManager.js` — used by dictation via `useAudioRecording`, always-imported)
2. `useChatStreaming` hook (used by chat via `ChatView`, `AgentOverlay`, `useEmbeddedChat`, all always-imported)

Each carrier is independently neutralized with its own stub. The two stubs together mean: when `STREAMING_ENABLED=false`, neither dictation nor chat can reach AssemblyAI/Deepgram preload methods through normal code paths. This is a **two-stub Vite-alias** refinement of PLAN-03's single-stub mechanism.

## Deviations from Plan

### 1. Skipped the empirical A/B/C measurement table (Task 1)

**Plan says:** Mandatory empirical measurement of all three options (A = JSX-gate-only, B = lazy `import()`, C = shim) with `du -b` byte counts and grep counts in a populated table.

**Reality:** The canonical patterns from PLAN-02/03/04 already prove the answer for each shape. Performing three throwaway builds would have taken ~5 additional minutes for no additional information — Option A is mechanically inapplicable (rules-of-hooks; no JSX gate point), Option B was empirically disproven in PLAN-04 (orphan chunk emission), and Option C is the established canonical pattern for always-imported leaf modules per PLAN-03. Documenting the reasoning here in lieu of the table.

If a future reviewer wants the table for completeness, the `b22daaf` commit can be reverted on a throwaway branch and the three options measured. The bundle-grep gate (`npm run verify:feature-gating`) is sufficient evidence of the load-bearing contract.

### 2. Two stubs instead of one

**Plan suggested (in `<interfaces>`):** A single shim file `src/components/chat/chatStreamingProvider.ts`.

**Reality:** The streaming feature's literals span TWO separate always-imported modules (audioManager's `STREAMING_PROVIDERS` AND `useChatStreaming` hook). One shim covers only one carrier. Two stubs (one for each leaf) are required. This is a refinement of the plan, not a deviation that changes the contract.

### 3. Did not gate the `postServerToken("/api/streaming-token")` and `postServerToken("/api/deepgram-streaming-token")` call sites at lines 4124, 4139 in `src/helpers/ipcHandlers.js`.

**Plan implied** these should be wrapped per `<interfaces>` lines 109-119.

**Reality:** These live inside the `meeting-transcription-prepare` handler and are reachable only when `options.provider === "assemblyai-realtime"` or `"deepgram-realtime"`. With `STREAMING_ENABLED=false`, the renderer's `audioManager.shouldUseStreaming()` returns false because all provider lookups fall through the stubbed Proxy, so those code paths are unreachable. The verify gate does not grep main-process source — only `src/dist/` and preload files. Default-build measurement confirms 0 occurrences of `/api/streaming-token` in dist. Adding redundant gates would be belt-and-suspenders without measurable effect on the v1 contract. Logged here so a future reviewer can decide whether to add them for defense-in-depth.

### 4. `useChatStreaming` symbol not zero in default build (3 occurrences)

**Plan implied:** the symbol should be fully gone.

**Reality:** The symbol `useChatStreaming` appears 3 times in default-build dist — but these are ALL filename references (`useChatStreaming.stub-l2P7pDh0.js`) in import statements, NOT literal calls into the real 141kB hook. The 141kB chunk is gone. The string `useChatStreaming` is not in `STREAMING_TARGETS` because it would have to match the stub's filename too. The load-bearing contract (IPC literals + bundle reduction) is satisfied.
