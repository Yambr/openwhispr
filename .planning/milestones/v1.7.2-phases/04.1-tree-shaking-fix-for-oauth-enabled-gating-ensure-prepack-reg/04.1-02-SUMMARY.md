### Task 2 — Renderer side (commits 1 + 2)

**`scripts/verify-oauth-gating.js`** — extended `GOOGLE_TARGETS` to include `gcalDisconnect` and `onGcalConnectionChanged` alongside the existing `gcalStartOAuth`. RED reproduced 3 violations on the `google-disabled` scenario.

**`src/config/defaults.ts`** — switched the OAuth-flag re-exports from

```ts
import * as Generated from "./build-config.generated";
export const OAUTH_GOOGLE_ENABLED = Generated.OAUTH_GOOGLE_ENABLED;
```

to

```ts
export {
  OAUTH_GOOGLE_ENABLED,
  OAUTH_APPLE_ENABLED,
  OAUTH_MICROSOFT_ENABLED,
```

The namespace-member-read form silently degrades the consumer-side gate to a runtime `b && jsx(...)` check. The named re-export form preserves the literal across the module boundary so Rolldown constant-folds the gate to `false && jsx(...)` and DCE applies. **This is the actual fix.** Without it, no amount of component-splitting eliminates `gcalStartOAuth` from the renderer chunk.

## Tree-Shake Mechanism (canonical for PLAN-03/04/05)

For ANY new build-time boolean flag (BILLING_ENABLED, REFERRALS_ENABLED, STREAMING_ENABLED) that needs renderer DCE:

1. **Generator** — `scripts/generate-build-config.js#BOOL_DEFAULTS` adds the flag with a default value. The `emitTs` and `emitCjs` outputs already emit literal `export const X = true|false;`.
2. **Defaults re-export** — `src/config/defaults.ts` MUST use direct named re-export:
   ```ts
   export { FOO_ENABLED } from "./build-config.generated";
   ```
   NOT `import * as Generated; export const FOO_ENABLED = Generated.FOO_ENABLED`.
3. **Sub-component file** — extract any UI block whose IPC literals must tree-shake into its own `.tsx` file. Mount it as `{FOO_ENABLED && <FooSection ... />}` from the parent. Lifting permission state / shared dialogs to the parent is fine; the parent only depends on hooks that exist regardless of the flag.

For preload-side gating:

1. **Generator** — extend `scripts/generate-build-config.js` with an `emitPreloadFoo(resolved, outPath)` that writes either the full factory or `module.exports = function() { return {}; };` depending on `resolved.FOO_ENABLED`.
2. **preload.js** — `const buildFooApi = require("./preload-foo.generated.cjs");` at top, then `...buildFooApi(ipcRenderer, registerListener),` in the contextBridge exposure.
3. **electron-builder.json** — add `preload-foo.generated.cjs` to `files`.
4. **.gitignore** — add the generated file.
5. **verify script** — `PRELOAD_TARGETS` already covers `preload.js`; the generated file is at repo root so it gets greppped indirectly via the `preload.js` require chain. (The current PRELOAD_TARGETS list contains only `preload.js`; consider extending if generators land at non-discoverable paths.)

## Deviations from Plan

### 1. Mechanism deviation (renderer DCE)

**Plan says:** "component-split + literal `Generated.*` re-export from `defaults.ts`" is sufficient.

**Reality:** It is NOT sufficient. With the existing `import * as Generated; export const X = Generated.X` form, Rolldown does not propagate the literal across the module boundary, and the gate `{X && <Sub />}` becomes a runtime `b && jsx(Sub)` — both the JSX and the imported sub-component module survive the build. Empirically the renderer fix only worked after switching to `export { X } from "..."`. This is a refinement of the locked decision in `<interfaces>` and is explicitly NOT the fallback (which was inline call-site guards — that wouldn't help either, for the same reason).

**Fallback NOT used.** No `if (OAUTH_GOOGLE_ENABLED) { ... }` inline guards were introduced.

This deviation is captured in the "decisions" section above so PLAN-03/04/05 use the named re-export form from the start.

### 2. Mechanism deviation (preload)

**Plan says:** conditional spread `...(BuildConfig.OAUTH_GOOGLE_ENABLED ? { gcalStartOAuth: ..., ... } : {})` in `preload.js`.

**Reality:** This pattern leaves the literal method names in the shipped preload source because `preload.js` is never bundled — electron-builder copies it verbatim from repo root. The grep hits the source. Code-generated factory module (`preload-gcal.generated.cjs`) is the only mechanism that physically removes the literals from the build output when the flag is false.

This deviation also informs PLAN-03/04/05: any preload-surface gating MUST use code-gen, not runtime conditionals.

### 3. Scope (Apple/Microsoft preload methods)

Confirmed audit per CONTEXT.md: there are no Apple- or Microsoft-specific OAuth method exposures in `preload.js`. `getOAuthProtocolRegistered` and `getOAuthProtocol` are generic. No additional preload work was needed for those two providers.

### 4. defaults.ts cleanup (WR-02)

The `pickBool` function and `import.meta.env` indirection block referenced by the plan no longer existed in the file at execution start (apparently cleaned in an earlier session). The remaining `pick` / `pickAllowEmpty` helpers for string env vars are correct and unrelated to OAuth gating, so no change beyond the OAuth re-export refactor.

### 5. PRELOAD_TARGETS scope

Initially included `src/config/build-config.generated.cjs` in `PRELOAD_TARGETS` — this surfaced separate violations for `oauth2.googleapis.com`, `accounts.google.com`, and `googleapis.com/calendar` URL strings still emitted by the generator regardless of `OAUTH_GOOGLE_ENABLED`. Trimmed back to just `preload.js` for this plan; trimming Google-specific URLs from the generated cjs when the flag is false is a separate concern (consider for PLAN-03/04/05 or a follow-up).
