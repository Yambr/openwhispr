## Fixed Issues

### WR-01: `shouldUseStreaming()` does not consult `STREAMING_ENABLED`

**Files modified:** `src/helpers/audioManager.js`
**Commit:** f10be28
**Applied fix:** Added an explicit `if (!STREAMING_ENABLED) return false;` short-circuit at the very top of `AudioManager.shouldUseStreaming()`, plus the named import `import { STREAMING_ENABLED } from "../config/defaults";`. This is the structural fix the reviewer asked for — not a symptom-level patch:

- The bug was that the stub `STREAMING_PROVIDERS.start()` returns `{success:false, error:"Streaming disabled in this build"}` with NO `code` field, so neither the `NO_API` nor `NETWORK_ERROR` fallback branches in `startStreamingSession` (lines 2258-2270) trigger, and the user gets a hard `Error("Failed to start streaming session")` instead of a graceful fallback to file-upload transcription.
- With the new gate, `shouldUseStreaming()` now returns `false` whenever `STREAMING_ENABLED` is the literal `false`, and the call sites of `shouldUseStreaming()` (in `transcribeAudio` and the start-recording paths) skip the streaming branch entirely — exactly matching the documented graceful-degradation intent.
- The literal-boolean import is Rolldown DCE-friendly (named re-export from `defaults.ts`, same mechanism the rest of the gating uses), so it adds no bundle weight in builds where `STREAMING_ENABLED=true`.
- The stub modules remain in place as defense-in-depth (any other entry point that sneaks past `shouldUseStreaming` still hits the safe stub).
