For each match: open the file at the line, confirm it's IN SCOPE per D-09 (one of the 5 buckets), confirm the value is HARDCODED (not already env-driven), and record it.

**Out of scope reminders (per D-09 — DO NOT inventory):**
- HuggingFace download URLs (e.g., `huggingface.co/.../resolve/main/...`)
- GitHub release URLs for sidecar binaries
- Build-tool URLs (npm registry, etc.)
- Local file:// URLs
- Documentation URLs in code comments
- HTTP URLs that are not API endpoints (e.g., links in error messages)

When in doubt, apply the **swap test** (per `<specifics>`): "Would a self-hoster swap this when pointing at their own backend?" If no → not in inventory.

**Step 2 — Compose the document.** Write `docs/CONFIG_INVENTORY.md` with this exact structure:

1. **`# Configuration Inventory`** — h1.

2. **Front matter** (3–5 sentences):
   - Purpose: catalogue every hardcoded value targeted for build-time replacement in Phase 3.
   - Scope: per D-09, the five CFG-01 buckets — backend URLs, OAuth client IDs, enterprise endpoints, default model registry overrides, LiteLLM-shaped URLs. Explicitly state out-of-scope items: HuggingFace/GitHub download URLs, build-tool URLs.
   - How to use: every row is a Phase 3 refactor target. The `proposed env-var` column is the *logical* name (`OPENWHISPR_*` prefix per D-12); renderer-side consumption sites need a `VITE_` prefix at consumption time, flagged in the `notes` column.
   - Cross-links: rows with backend/enterprise/litellm category link to `docs/BACKEND_SPEC.md` endpoint cards in `notes`; oauth rows link to `docs/OAUTH_SPEC.md` provider cards (per D-13).

3. **Optional summary table** (Claude's Discretion — INCLUDE one): row counts per category, e.g.:

   | Category | Count |
   |----------|-------|
   | backend | N |
   | oauth | N |
   | enterprise | N |
   | model-registry | N |
   | litellm | N |
   | **Total** | N |

4. **`## Inventory`** — single 5-column table per D-11:

   ```markdown
   | file:line | current value | proposed env-var | category | notes |
   |-----------|---------------|------------------|----------|-------|
   | src/config/constants.ts:116 | https://api.openwhispr.com | OPENWHISPR_BACKEND_URL | backend | Already locked by REQ CFG-04. Used in 3 call sites — consolidate via single export. Renderer + main; needs VITE_ prefix at renderer consumption. See [BACKEND_SPEC §Auth](BACKEND_SPEC.md#auth). |
   | ... | ... | ... | ... | ... |
   ```

   **Row ordering (Claude's Discretion per D-13):** Order by category first (backend → oauth → enterprise → model-registry → litellm), then alphabetically by file path within each category. Apply consistently.

   **Naming rules per D-12:**
   - `OPENWHISPR_*` prefix on every var
   - Logical kebab→snake meaning (e.g., `OPENWHISPR_OAUTH_GOOGLE_CLIENT_ID`)
   - Boolean OAuth gates (Phase 4 CFG-03): `OPENWHISPR_OAUTH_<PROVIDER>` (e.g., `OPENWHISPR_OAUTH_GOOGLE` = boolean enable/disable)
   - Enterprise: `OPENWHISPR_ENTERPRISE_<SERVICE>_<DETAIL>` (e.g., `OPENWHISPR_ENTERPRISE_BEDROCK_REGION`, `OPENWHISPR_ENTERPRISE_AZURE_OPENAI_URL`)
   - LiteLLM: `OPENWHISPR_LITELLM_<USE>_URL` (e.g., `OPENWHISPR_LITELLM_TRANSCRIPTION_URL`)
   - The locked `OPENWHISPR_BACKEND_URL` (CFG-04) is the naming anchor.

   **Notes column conventions:**
   - Flag renderer-side consumption: `Renderer; needs VITE_ prefix at consumption.` or `Main process only.` or `Both; needs VITE_ at renderer site.`
   - Flag multi-callsite values: `Used in N call sites — consolidate to single export.`
   - Cross-link to Phase 1 docs where applicable: `See [BACKEND_SPEC §<anchor>](BACKEND_SPEC.md#<anchor>)` or `See [OAUTH_SPEC §<provider>](OAUTH_SPEC.md#<provider>)`.
   - For OAuth rows that share infrastructure (Apple/Microsoft tunnel through OpenWhispr cloud per Phase 1 D-04): give each provider its own row even if they share a sign-in shim, with notes flagging the shared backend.
   - Per D-10: do NOT annotate LiteLLM rows with v2 relevance.

5. **`## Verification Notes`** (short — 3–5 lines): briefly state how a developer should verify the inventory is complete before starting Phase 3 — re-run the discovery greps from Step 1 and check every match is either (a) in the table, (b) explicitly out of scope per the documented exclusions.

**Quality bar:**
- Every row's `file:line` must point to a real current line in the source (not a stale Phase 1 reference).
- No row may use a placeholder env-var name (e.g., `OPENWHISPR_TODO`) — every name must be the final proposed name.

- Empty categories (e.g., if there are zero `model-registry` overrides) should still appear in the summary table with count `0` and a one-line note in the inventory table area: `_No entries — model registry has no in-source URL overrides._`
