# v1.7.2 Cross-Phase Integration Check

Sister-document to `.planning/v1.7.2-MILESTONE-AUDIT.md` (commit `ddde964b`).
That audit answered "are the per-phase requirements satisfied?". This one
answers "do the phases compose end-to-end, and is the wiring solid enough
to build v1.8.0 onto?".

## 1. Executive verdict: GO to close v1.7.2 and start v1.8.0

GO with eyes open. v1.7.2 is shippable: gates compose, lockdown wires through
renderer+preload+main consistently, realtime language pipeline is green-tested
through the WSS URL construction layer, and the corporate-minimal release.yml
binary builds and runs (live-verified per `v178_prod_live_results.md`).

But v1.8.0 cannot just *add* a "Server URL" field on top of HEAD — it needs
two surgical refactors as part of its dispatch, because two pre-existing
patterns directly conflict with runtime URL reconfiguration:

1. The renderer's HTTP "backend" host today is `OPENWHISPR_API_URL` (from
   `VITE_OPENWHISPR_API_URL`), not `OPENWHISPR_BACKEND_URL`. Two parallel
   env-var systems carrying the same semantic happen to be kept in sync by
   `.github/workflows/release.yml:114-117,128-131,…` (sets BOTH). v1.8.0
   should collapse to one SoT before adding runtime override.
2. `src/lib/auth.ts:12` calls `createAuthClient({ baseURL: AUTH_URL })` at
   module-load time. `AUTH_URL` is `import.meta.env.VITE_AUTH_URL`, a frozen
   build-time literal. There is no path today to point a new request at a
   different host without re-importing the module. v1.8.0's "user edits URL
   on onboarding" needs either (a) a factory pattern so authClient is
   re-created on URL change, or (b) a thunk-style baseURL.

Both are bounded refactors, not unknowns — but they are real work, not
"already done."

### C. Better Auth client re-initializable / accepts URL dynamically — **FAIL**

| Check | Evidence | Result |
|---|---|---|
| `authClient` is a module-level singleton | `src/lib/auth.ts:12-25` (`export const authClient = createAuthClient({ baseURL: AUTH_URL, … })`) | FAIL — frozen at first import |
| `AUTH_URL` is a runtime constant | `src/lib/auth.ts:11` (`import.meta.env.VITE_AUTH_URL` — substituted at build, not at runtime) | FAIL — build-time only |
| Renderer URL constant is also frozen | `src/config/constants.ts:116` (`OPENWHISPR_API_URL = env.VITE_OPENWHISPR_API_URL` at module load) | FAIL |
| Main-process `getApiUrl()` reads env fresh each call | `src/helpers/ipcHandlers.js:3387-3391` (function, called per-handler) | PARTIAL — could be repointed at runtime, but reads `process.env`, not a renderer-supplied value |
| Better Auth supports dynamic baseURL | Not investigated (would need to check `better-auth/react` API docs); the API surface used is `signUp.email`, `signIn.email`, `signIn.social`, `signOut`, `sendVerificationEmail`, `requestPasswordReset`, `getSession` (via header callback) | UNKNOWN |
| `OPENWHISPR_API_URL` consumers (renderer side: `AuthenticationStep.tsx`, `EmailVerificationStep.tsx`, `auth.ts:deleteAccount`) all construct URL fresh per fetch using the module-load constant | YES | FROZEN |

**Verdict: FAIL.** This is the hardest v1.8.0 prereq. Options for v1.8.0:

1. **authClient factory pattern.** Convert `export const authClient = …` to
   `let authClient = createAuthClient(…); export function getAuthClient() { return authClient; }`,
   add `function reinitAuthClient(newBaseURL) { authClient = createAuthClient({ baseURL: newBaseURL, … }); }`,
   and call this from the onboarding URL-save handler. Renderer-wide
   refactor: `~30` import sites need to switch from `authClient` named
   import to `getAuthClient()` call.

2. **Module-level let + re-export.** Less invasive: `export let authClient = …`
   and mutate in place. Risky if consumers destructure or store references.

3. **Thunk baseURL.** Some Better Auth versions accept a `baseURL: () => string`
   getter — verify against `better-auth/react` docs. If supported, this is the
   cleanest path: one line in `auth.ts:12-13` changes from
   `baseURL: AUTH_URL` to `baseURL: () => getRuntimeAuthUrl()`.

The same shape applies to `OPENWHISPR_API_URL` (consumed in 3 renderer files
+ ipcHandlers). Either collapse it into a getter or wire it through a Zustand
store that other components subscribe to.

**Recommendation:** investigate `better-auth/react`'s `baseURL` thunk support
in Phase 0 of v1.8.0. If it accepts a function, do that. Otherwise factor
the singleton.

## Bottom line

v1.7.2 is **closeable** — gates compose, lockdown works end-to-end, realtime
language pipeline is green-tested, e2e suite is healthy. There are no
blockers.

But v1.8.0's "Custom Server URL onboarding" lands on a base where:
- The HTTP backend host has two parallel env vars (INT-02) that need
  collapsing before the runtime override is wired
- The Better Auth client is a frozen module singleton (INT-01) that
  cannot be repointed at runtime without refactor

Both are bounded, well-understood, and should be the first two phases of
v1.8.0 — not surprises that emerge mid-milestone.
